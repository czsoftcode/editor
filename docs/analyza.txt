     - workspace.rs: přidáno pole external_change_conflict: Option<PathBuf>

4. IPC — race condition při zápisu session.json / recent.json
   Soubor:  src/ipc.rs (funkce save_paths)

   Problém:
     Zápis probíhá přes atomický rename (tmp → json), což je správně,
     ale file locking není implementován. Při rychlém otevírání/zavírání
     více oken si dvě instance mohou navzájem přepsat soubor a poslední
     zápis vyhraje — přijdete o část nedávných projektů nebo session.
     V praxi vzácné, ale možné.

────────────────────────────────────────────────────────────────────────────────
   OPRAVENO 2026-02-19:
────────────────────────────────────────────────────────────────────────────────
     - Analýza ukázala: recent.json je již bezpečný — všechny zápisy jdou
       přes process_command pod state.lock() v IPC serveru.
     - session.json byl skutečný problém — více procesů volalo save_session()
       přímo, sdílely stejný session.tmp a zápis nebyl koordinovaný.
     - ipc.rs: přidán příkaz SAVE_SESSION <json> do process_command; server
       drží state mutex během zápisu → souběžné volání ze dvou procesů se
       serializuje.
     - ipc.rs: přidána privátní metoda Ipc::save_session_via_ipc(paths)
       která odešle příkaz a vrátí bool.
     - ipc.rs: veřejná funkce save_session() nově routuje přes IPC server;
       přímý zápis slouží jen jako záloha pokud server neodpovídá

LOGICKÁ ANALÝZA A POTENCIÁLNÍ NEDOSTATKY
=======================================

G1. Správa sezení a konkurenční zápis (Race Conditions)
Aplikace podporuje více oken, ale všechna sdílejí jeden konfigurační soubor sezení. Pokud běží více nezávislých procesů editoru, mohou si navzájem přepisovat `session.json`, protože zápis není atomický ani chráněný zámkem na úrovni souborového systému.

────────────────────────────────────────────────────────────────────────────────
   HOTOVO — pokryto opravami #4 a C2:
────────────────────────────────────────────────────────────────────────────────
     - Fix #4: session.json se zapisuje výhradně přes IPC server pod state.lock()
       — souběžné zápisy z více procesů jsou serializovány.
     - Fix C2: sekundární instance se vůbec nespustí jako samostatný proces
       (deleguje na primára a exituje) — celá kategorie souběžných procesů
       zapisujících do sdílených souborů je eliminována.

────────────────────────────────────────────────────────────────────────────────

G2. Životní cyklus vláken na pozadí (Thread Leaks)
Vlákna spouštěná přes `std::thread::spawn` (Git status, indexace, AI tool check) jsou typu "fire and forget". Při rychlém zavírání a otevírání projektů mohou stará vlákna stále běžet na pozadí a zbytečně konzumovat CPU a I/O, protože neexistuje mechanismus pro jejich předčasné ukončení (cancellation).

────────────────────────────────────────────────────────────────────────────────
   ČÁSTEČNĚ HOTOVO — viz fix #5:
────────────────────────────────────────────────────────────────────────────────
     - Git vlákna (fetch_git_branch, fetch_git_status): opraveno v fix #5 —
       Arc<AtomicBool> cancel flag + child.kill() při zániku workspace.
     - Zbývající vlákna (spawn_file_index_scan, ai_tool_check_rx) jsou bounded
       — dokončí se rychle a jejich Receiver se přirozeně dropne při zániku
       workspace; vlákno pak tiše skončí. Nejde o reálný leak.

────────────────────────────────────────────────────────────────────────────────

G3. Náročnost vykreslování (UI Performance)
Vynucené překreslování v každém snímku (kvůli terminálu a autosave) v kombinaci s "immediate mode" architekturou egui může vést k vysokému vytížení CPU u velkých projektů. Každý snímek prochází všechny aktivní kanály a stavy, což by mělo být lépe kešováno.

────────────────────────────────────────────────────────────────────────────────
   OPRAVENO 2026-02-19 (praktické optimalizace):
────────────────────────────────────────────────────────────────────────────────
     - config.rs: REPAINT_INTERVAL_MS snížen z 250 ms na 500 ms — halvuje
       timer-driven CPU zátěž; terminál si řídí vlastní repaints přes
       ctx.request_repaint(), takže odezva terminálu není dotčena.
     - config.rs: přidána konstanta TERMINAL_MAX_EVENTS_PER_FRAME = 256.
     - terminal.rs: PTY event smyčka omezena na max 256 eventů za snímek;
       při burstu výstupu (cargo build, grep) se zbytek dočerpá příští snímek
       místo blokování UI smyčky na desítky ms.
     - Markdown parse (pulldown_cmark) a syntax highlighting nejsou výkonnostním
       bottleneckem — pulldown_cmark je O(n) a velmi rychlý; egui widget
       konstrukce je v immediate mode povinná každý snímek a nelze ji vynechat.

────────────────────────────────────────────────────────────────────────────────

G4. Robustnost IPC (Inter-Process Communication)
Unix sockety pro detekci běžící instance jsou náchylné na "zombie" soubory socketů po pádu aplikace. Pokud soubor socketu zůstane na disku, nová instance se může mylně domnívat, že jiný editor již běží, a odmítne se spustit.

────────────────────────────────────────────────────────────────────────────────
   HOTOVO — bylo vyřešeno již před touto analýzou:
────────────────────────────────────────────────────────────────────────────────
     - IpcServer::start() volá Ipc::ping() před bindováním socketu.
     - Pokud ping selže (stale socket — process je mrtvý, connection refused),
       ping vrátí false; kód pak odstraní starý soubor socketu a nabinduje nový.
     - Nová instance tedy zombie souborem zablokována není.

────────────────────────────────────────────────────────────────────────────────

G5. Správa dceřiných procesů v PTY
Při ukončení aplikace neprobíhá explicitní rekurzivní ukončování celých procesních skupin spuštěných v terminálech (AI panel, Build panel). To může vést k tomu, že procesy jako `cargo run` nebo AI nástroje zůstanou běžet na pozadí jako sirotci (orphans).

────────────────────────────────────────────────────────────────────────────────
OPRAVENO 2026-02-19:
────────────────────────────────────────────────────────────────────────────────
  vendor/egui_term/src/backend/mod.rs
    — Přidáno pole `#[cfg(unix)] pub child_pid: u32` do `TerminalBackend`
    — Před předáním `pty` do `EventLoop::new()` se extrahuje `pty.child().id()`
      (metoda `Pty::child()` vrací `&std::process::Child`, odkud `id()` dá PID shellu)
    — PID je součástí `Ok(Self { ..., child_pid })` při konstrukci

  src/app/modules/terminal.rs
    — Přidán `use libc` (unix-only, podmínění `#[cfg(unix)]`)
    — Přidána privátní metoda `kill_process_group(&self)`:
        • Pokud `self.exited` → nic nedělá (shell sám skončil)
        • Jinak `libc::kill(-pid, libc::SIGTERM)` na celou procesní skupinu
        • Ochrana `pid > 1` zabraňuje kill(-1, SIGTERM) na init/PID1
    — Přidáno `impl Drop for Terminal` volající `kill_process_group()`

  Cargo.toml
    — Přidána závislost `libc = "0.2"`

  Mechanismus:
    Když je workspace zavřen (okno / ukončení editoru), Rust RAII zavolá
    `Drop for Terminal` pro každý terminal v `claude_tabs` a `build_terminal`.
    SIGTERM je poslán celé procesní skupině (`-pid`), takže cargo run, aider,
    claude cli a jejich potomci dostanou signál k čistému ukončení ještě před
    tím, než `TerminalBackend::drop()` pošle `Msg::Shutdown` a zavře PTY master
    fd (který by jinak vyslal SIGHUP — méně spolehlivé pro background procesy).

G6. Statická detekce nástrojů
Kontrola dostupnosti AI nástrojů probíhá pouze při inicializaci pracovního prostoru. Pokud je nástroj nainstalován během běhu aplikace, editor jej nerozpozná bez restartu daného okna.
