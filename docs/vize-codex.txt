Vize PolyCredo Editoru na 12 mesicu (expanze-ready)
===================================================

Datum: 2026-02-19

1) Cehova vize za rok
---------------------
Za 12 mesicu by mel byt PolyCredo Editor vnimany jako:
- rychly desktop editor/IDE pro Rust + web stack,
- AI-native prostredi pro vyvojare, ne jen "terminal s AI v panelu",
- stabilni produkt pripraveny na rust uzivatelu, vice platforem a plugin ekosystem.

Strategicky cil:
Prejit z "silneho lokalniho editoru" na "platformu", kde se daji bez velkeho refaktoru pridavat:
- nove jazyky,
- nove AI providery,
- tymove funkce,
- enterprise pozadavky (audit, policy, sprava).

2) Produktove pilire
--------------------
P1: Spolehlive jadro editoru
- Zero-panic policy v runtime (recoverable chyby, zadne expect v kritickych cestach).
- Predikovatelne chovani u souboru, watcheru, session a recover flow.
- Stabilni startup a obnoveni session i na velkych projektech.

P2: AI jako prvotridni vrstva
- Jednotne AI API uvnitr aplikace (provider-agnostic vrstva).
- Kontextove operace nad projektem: explain, refactor, fix, test, commit summary.
- Bezpecnostni brzdy: potvrzeni pred destruktivni zmenou, audit trail AI akci.

P3: Vyvojarska produktivita
- Rychly file/search index (incremental, background).
- Lepsi build/test loop (diagnostika, quick-fix navrhy, one-click rerun).
- Command palette + workflow automatizace (makra/commands).

P4: Expanze a adopce
- Lokalizace (cs/en uz hotovo, rozsireni na dalsi jazyky).
- Dokumentace "getting started", "team setup", "extension guide".
- Predpriprava na plugin ekosystem a komunitni prispevky.

3) Jak ma vypadat aplikace za rok (konkretni obraz)
----------------------------------------------------
Uzivatel otevre projekt a do 2-3 sekund ma:
- naceteny workspace,
- aktualni git stav,
- pripraveny file picker a project search index,
- AI asistenta, ktery zna kontext projektu bez manualniho kopirovani.

V beznem dni umi:
- psat kod bez lagu i ve velkem repozitari,
- spoustet build/test z jednoho mista s jasnou diagnostikou,
- vyuzivat AI na lokalni ulohy i vicekrokove flow,
- vratit se po restartu presne do stejneho stavu.

Pro team leada/firmu:
- jasny release rytmus,
- meritelnou kvalitu,
- predikovatelny provoz bez "nahodnych zamrznuti".

4) Architektonicke smerovani pro expanzi
----------------------------------------
A) Internal platform vrstvy
- Zavest "service boundary" uvnitr app:
  - `core` (editor/session/io),
  - `indexing` (souborovy a textovy index),
  - `ai` (provider adaptery),
  - `integrations` (git/build/test).
- Sjednotit event model (interne event bus / message typy) misto ad-hoc handoffu.

B) Plugin-ready foundation
- Definovat stabilni extension body:
  - command registry,
  - panel registry,
  - action hooks (on_save, on_build, on_open_file).
- Prvni faze muze byt "internal plugins" (kompilovane moduly), az potom externi ABI.

C) Data a indexace
- Drzet persistentni index projektu (inkrementalni update z watcheru).
- Pridat metadata vrstvu pro rychle dotazy (soubor, jazyk, symbol, posledni zmeny).

D) Operational readiness
- Telemetrie (opt-in) + crash reporting + anonymizovane vykonove metriky.
- Diagnosticky balik pro support (verze, OS, logs, posledni chyby).

5) Plan po kvartalech (12 mesicu)
---------------------------------
Q1 (mesice 1-3): Stabilita a zaklady
- Dokoncit vsechny otevrene audit fixy s testy.
- Build pipeline quality gate (`fmt`, `clippy -D warnings`, `check`, `test`).
- Odstranit hlavni zdroje UI blokovani a panik.

Q2 (mesice 4-6): Vykon a workflow
- Sdileny project index pro Ctrl+P + project search.
- Command palette (MVP) + sjednoceni nejcastejsich akci.
- Build/test UX: lepsi diagnostika + rychle navigace.

Q3 (mesice 7-9): AI vrstva 2.0
- Provider abstraction + jednotny prompt/context pipeline.
- AI akce primo na soubor/tab/workspace s transparentnim diffem.
- Policy guardrails (potvrzeni, omezeni scope, audit log AI zasahu).

Q4 (mesice 10-12): Expanze
- Plugin architecture alpha (internal plugin model).
- Team-ready funkce (workspace profily, sdilene preset flow, export/import nastaveni).
- Release channels (stable/beta/nightly) + release notes automatizace.

6) Co je potreba zmenit mimo kod
--------------------------------
- Produktove rizeni:
  - jednotny roadmap board (must/should/could),
  - kazdy release ma 1-2 hlavni tema, ne "vsechno najednou".
- QA:
  - regressions checklist pro editor/ipc/watcher/search.
  - smoke test scenare pro Linux/Wayland/X11 (a pozdeji Win/macOS).
- Dokumentace:
  - docs pro uzivatele,
  - docs pro prispivatele,
  - changelog s dopadem na workflow.

7) KPI a metriky uspechu
------------------------
Technicke:
- Crash-free sessions >= 99.5 %.
- Cold start median < 2.5 s.
- Ctrl+P response median < 120 ms na strednim projektu.
- Search in project first result < 400 ms (cache warm).

Produktove:
- D7 retention roste mesic po mesici.
- Podil aktivnich uzivatelu AI panelu > 50 %.
- Mean time to first successful build v novem projektu klesa.

Procesni:
- Release cadence: minimalne 1 stabilni release / mesic.
- 90 %+ PR projde quality gates bez opakovaneho fix kola.

8) Rozhodnuti, ktera drzet od zacatku
-------------------------------------
- Nepoustet se do velkeho refaktoru bez jasne business hodnoty.
- Drzet "small safe increments": male patche, meritelny dopad, rychla verifikace.
- Expanzi stavet na stabilnim jadru: nejdriv robustnost a observabilita, pak feature akcelerace.

Zaver
-----
Za rok muze byt PolyCredo Editor realne pripraveny na expanzi, pokud se udrzi disciplina:
1) stabilita a kvalita jako zaklad,
2) vykon a workflow jako diferencni vyhoda,
3) AI + plugin vrstva jako multiplikator rustu.
