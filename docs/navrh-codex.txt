Návrh dalšího postupu a vylepšení aplikace PolyCredo Editor
==========================================================

Datum: 2026-02-19
Podklady: průchod kódu v `src/`, `AUDIT-2026-02-18.md`, `AUDIT-2026-02-19.md`, `docs/architecture.txt`

1) Shrnutí aktuálního stavu
---------------------------
- Architektura je už teď solidní: single-process multi-window, oddělené moduly (`workspace`, `ipc`, `watcher`, `editor`, `terminal`).
- Většina priorit z auditu 2026-02-18/19 je už zapracovaná (validace cest v IPC, JSON přes `serde_json`, watcher remove eventy, asynchronní operace v dialogu/wizardu, limity IPC workerů).
- Aktuálně je ale rozbitý build: `cargo check` padá na chybě `E0061` (`src/app/modules/editor/mod.rs` volá `search_bar` bez `i18n` argumentu).

Doporučení: nejdřív vrátit projekt do "green" stavu, až pak pokračovat ve funkcích.

2) Doporučená roadmapa (pořadí)
--------------------------------

Fáze 0: Stabilizační minimum (0-1 den)
- Opravit compile blocker (`search_bar(ui, i18n)` vs. aktuální signatura).
- Spustit a zaznamenat baseline: `cargo check`, `cargo test`.
- Zajistit, že tento baseline bude povinný před každým mergem (lokálně + CI).

Výstup fáze:
- "Build green" na aktuálním `main`.
- Krátký changelog "co bylo rozbité / co je opravené".

Fáze 1: Robustnost a bezpečnost hranic (1-2 týdny)
- IPC hardening:
  - při dosažení limitu workerů vracet explicitní odpověď (např. `ERR busy`) místo tichého dropu spojení,
  - sjednotit logování všech IPC chyb jedním helperem,
  - zvážit kontrolu oprávnění socket souboru (jen aktuální uživatel).
- Path policy:
  - sjednotit canonicalizaci a validaci cest do jednoho sdíleného helperu,
  - doplnit testy na alias cesty (`/a/b`, `/a/./b`, symlink varianty).
- Error visibility:
  - user-facing I/O chyby všude propagovat do toastu,
  - interní chyby minimálně logovat konzistentním formátem.

Výstup fáze:
- Méně "tichých" selhání, snadnější debug.

Fáze 2: Výkon velkých projektů (2-4 týdny)
- Jeden sdílený index souborů pro:
  - Ctrl+P,
  - hledání v projektu,
  - případně rychlé filtry ve file tree.
  (Dnes se část práce stále dubluje a opakovaně čte FS.)
- Incremental update indexu přes `ProjectWatcher` místo plného re-scanu.
- Limit a filtry pro projektové hledání:
  - skip binárních souborů,
  - soft-limit velikosti souboru,
  - debounce/cancel při rychlém zadávání dotazu.

Výstup fáze:
- Lepší odezva na velkých repozitářích, méně špiček CPU/I/O.

Fáze 3: UX a produktové vylepšení (4+ týdnů)
- Command palette (`Ctrl+Shift+P`) pro sjednocení akcí.
- Lepší práce s chybami buildu:
  - rychlé filtry,
  - přesnější jump-to-location,
  - možnost zkopírovat diagnostiku.
- Session obnova včetně kontextu:
  - aktivní tab,
  - caret pozice,
  - scroll pozice,
  - rozbalení file tree.
- Nastavení AI panelu oddělit na per-workspace vs. globální profil.

Výstup fáze:
- Vyšší uživatelská efektivita a menší počet kliků na běžné úkony.

3) Konkrétní backlog (doporučené pořadí patchů)
------------------------------------------------
1. Oprava `cargo check` blockeru (`search_bar` call-site).
2. Testy pro Unicode vyhledávání + regrese na char boundaries.
3. IPC `ERR busy` + jednotné logování IPC operací.
4. Sjednocení path-normalization helperů a jejich testy.
5. Shared file index service (Ctrl+P + project search).
6. Incremental refresh indexu z watcher eventů.
7. Command palette (MVP).

4) Kvalitativní brány (doporučeno zavést hned)
-----------------------------------------------
- Povinné před merge:
  - `cargo fmt --check`
  - `cargo clippy -- -D warnings`
  - `cargo check`
  - `cargo test`
- U kritických oblastí (`ipc`, `search`, `watcher`) přidat minimálně regresní test při každém fixu.

5) Poznámka k rozsahu
----------------------
Dokud nebude stabilizovaná Fáze 0 a 1, nedoporučuji velký refaktor architektury.
Lepší je držet malé cílené patche s rychlou verifikací, protože aktuální modulární rozdělení je dostatečně dobré a větší přestavba by zvýšila riziko regresí.
