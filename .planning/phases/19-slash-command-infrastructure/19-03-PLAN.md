**Change 1: Track code-fence state in render_markdown (lines 46-93)**

Add a `in_code_fence: bool` variable initialized to `false` before the `for line in text.lines()` loop. Inside the loop, before any other processing, check if `trimmed.starts_with("```")`. If so, toggle `in_code_fence`. When `in_code_fence` is true, skip all monologue detection logic — just append the line to `current_block` as-is. This prevents premature flushing that would split a code block across multiple CommonMarkViewer calls.

```rust
let mut in_code_fence = false;

for line in text.lines() {
    let trimmed = line.trim();

    // Track fenced code blocks — skip all monologue detection inside them
    if trimmed.starts_with("```") {
        in_code_fence = !in_code_fence;
        current_block.push_str(line);
        current_block.push('\n');
        continue;
    }

    if in_code_fence {
        current_block.push_str(line);
        current_block.push('\n');
        continue;
    }

    // ... existing monologue detection logic unchanged ...
}
```

**Change 2: Skip path regex inside code blocks in flush_block (lines 118-131)**

In `flush_block`, before applying the `path_re` regex, check if the block contains a fenced code block. If the text contains triple backtick lines, skip regex replacement entirely for that block to preserve code block content.

Replace the existing path_re application block with:

```rust
if let Some(re) = &path_re {
    // Do NOT replace paths inside fenced code blocks — it breaks code block syntax
    let has_code_fence = text.lines().any(|l| l.trim().starts_with("```"));
    if !has_code_fence {
        text = re
            .replace_all(&text, |caps: &regex::Captures| {
                if caps.name("link").is_some() {
                    caps[0].to_string()
                } else if let Some(c) = caps.name("code_inner") {
                    format!("[{}](code)", c.as_str())
                } else {
                    format!("[{}](path)", &caps[0])
                }
            })
            .to_string();
    }
}
```

This is a conservative approach: if a block contains ANY code fence, skip all path replacement for that block. This is safe because code blocks are typically flushed as a single block, and path links inside code blocks are undesirable anyway.
