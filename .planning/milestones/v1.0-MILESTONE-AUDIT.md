### Cross-Phase Wiring Architecture

```
Phase 1 (settings.rs)
  LightVariant + to_egui_visuals() + syntect_theme_name() + apply()
       │
       ▼
Phase 2 (terminal/instance/theme.rs + git_status.rs)
  terminal_theme_for_visuals() ← ui.visuals() (result of apply())
  git_color_for_visuals() ← ui.visuals()
       │
       ▼
Phase 3 (settings modal + tone_light_palette + warm_ivory_bg)
  picker writes → settings_draft → settings_version bump → Phase 1 apply() triggers
  tone_light_palette() extends Phase 2 terminal path with per-variant blending
  git_color_for_visuals() reads panel_fill from Phase 1 visuals (variant-aware by construction)
```

`settings_version` AtomicU64 je centrální integrační sběrnice — odděluje write side (Settings modal, discard flow) od read side (root + deferred viewport update loop).

## Tech Debt Summary

### Manuální GUI verifikace (doporučená, neblokující)

Fáze 03 identifikuje 4 body které nelze ověřit staticky:

1. **3 karty vedle sebe** — layout horizontal_wrapped závisí na skutečné šířce modalu
2. **WarmIvory terminál krémový tón** — percepce jemného tónálního rozdílu vyžaduje lidské zrakové hodnocení
3. **Live preview multi-window** — synchronizace deferred viewportů nelze simulovat bez běžícího GUI
4. **Cancel → okamžitý návrat** — vizuální verifikace cancel flow

Toto tech debt neblokuje milestone — automatizované testy pokrývají correctness, lidská verifikace pokryje percepci.

### Architektonická poznámka

Deferred viewport settings apply nastane jeden frame po Cancel (existující design s `settings_version` gate). Není bug, je to záměrný kompromis mezi jednoduchou architekturou a immediate-mode UI modelem egui.
