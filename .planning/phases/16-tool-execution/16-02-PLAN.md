  <files>src/app/ai/ollama.rs</files>
  <behavior>
    - parse_ndjson_line with tool_calls JSON -> StreamEvent::ToolCall { id, name, arguments }
    - parse_ndjson_line with regular token -> StreamEvent::Token (unchanged)
    - parse_ndjson_line with done=true -> StreamEvent::Done (unchanged)
    - Tool call ID generated as "tc_{name}_{counter}" format
    - stream_chat includes "tools" array in request body when tools Vec is non-empty
    - Message serialization includes tool_calls for assistant messages that have them
    - Message serialization includes role="tool" for tool result messages
    - capabilities() returns supports_tools: true
  </behavior>
  <action>
    Modify `src/app/ai/ollama.rs`:

    1. **Update capabilities():** Change `supports_tools: false` to `supports_tools: true`.

    2. **Extend AiProvider trait signature (in provider.rs):** Add `tools: Vec<AiToolDeclaration>` parameter to `stream_chat()`. Update the trait:
       ```rust
       fn stream_chat(&self, messages: Vec<AiMessage>, config: ProviderConfig, tools: Vec<AiToolDeclaration>) -> mpsc::Receiver<StreamEvent>;
       ```
       Update the single call site in `logic.rs` to pass `get_standard_tools()`.

    3. **Update stream_chat() in ollama.rs:**
       - Convert `tools` Vec into Ollama format:
         ```json
         [{"type": "function", "function": {"name": "...", "description": "...", "parameters": {...}}}]
         ```
       - Add `"tools": tools_json` to request body (only if tools is non-empty).
       - Per research recommendation: use `"stream": false` when tools are present (simpler, more reliable across models). When tools are empty, keep `"stream": true`. This means when tools are present, read full response instead of NDJSON lines, parse tool_calls or content, emit as StreamEvent.
       - For non-streaming tool response: parse full JSON, check for tool_calls array, emit StreamEvent::ToolCall or StreamEvent::Token + StreamEvent::Done.

    4. **Update message serialization** in stream_chat and send_chat:
       - For messages with `tool_call_name.is_some()` and `role == "assistant"`: serialize with `tool_calls` array (Ollama format).
       - For messages with `role == "tool"`: serialize as `{"role": "tool", "content": "..."}`.
       - For messages with `tool_call_arguments.is_some()`: include in tool_calls serialization.

    5. **Update parse_ndjson_line()** to also detect tool_calls (for future streaming tool support):
       - Before checking content, check `parsed["message"]["tool_calls"]`
       - If tool_calls array is non-empty, take first entry, extract `function.name` and `function.arguments`
       - Generate local ID: use a static AtomicU32 counter — `format!("tc_{}_{}", name, COUNTER.fetch_add(1, Ordering::Relaxed))`
       - Return `StreamEvent::ToolCall { id, name, arguments }`

    6. **Add tests** for:
       - parse_ndjson_line with tool_calls JSON
       - tools_json serialization format
       - Message serialization for tool role messages
