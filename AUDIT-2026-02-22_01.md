## KRITICKÉ PROBLÉMY

### 1. Unwrap na Mutex locks — riziko paniky
**Soubory:** `src/app/mod.rs`, `src/app/lsp/mod.rs`, `src/app/ui/background.rs`, `src/ipc.rs`
**Výskyt:** 50+ míst

Kód používá `.lock().unwrap()` na `Arc<Mutex<>>` strukturách bez ochrany. Pokud vlákno, které drží zámek, panickuje, ostatní vlákna se zablokují a aplikace havaruje s "poisoned mutex" chybou.

```rust
// src/app/mod.rs:156
let mut s = shared.lock().unwrap();

// src/app/lsp/mod.rs:86
let mut diag_map = diagnostics_handler.lock().unwrap();

// src/app/ui/background.rs:92
let auto_show = shared.lock().unwrap().settings.auto_show_ai_diff;
```

**Doporučení:**
- Nahradit `.lock().unwrap()` za `.lock().expect("popis kontextu")` pro lepší diagnostiku
- Zvážit `.try_lock()` s fallback logikou na místech s nízkou latencí
- Minimalizovat dobu držení zámku (scope bracketing)

### 2. Race condition v sandbox staged files
**Soubor:** `src/app/ui/background.rs`
**Řádky:** 73–79, 84–101

`sandbox_staged_dirty` se nastavuje a čte bez garantované atomicity. Sekvence:

1. Frame N: `sandbox_staged_dirty = true` se nastaví při změně souboru
2. Frame N+1: `refresh_sandbox_staged_cache_if_due()` resetuje flag a spouští async sken
3. Pokud přijde další change event mezi resetem a dokončením skenu, změna se ztratí

```rust
// src/app/ui/background.rs:77–79
if change.path().starts_with(sandbox_root) {
    ws.sandbox_staged_dirty = true;
}
```

**Doporučení:**
- Použít epoch/generation čítač místo boolean dirty flagu
- Atomicky párovat check + spawn operaci

### 3. Absence ochrany před duplicitním git procesem
**Soubor:** `src/app/ui/background.rs`
**Řádky:** 210–219

Git refresh se spouští každých 15 sekund, ale neexistuje kontrola, zda předchozí git proces stále běží. Při pomalém I/O nebo blokování disku se procesy mohou hromadit.

```rust
if ws.git_last_refresh.elapsed().as_secs() >= 15 {
    // Spouští nový git status bez kontroly, zda předchozí skončil
```

Kontrola `ws.git_status_rx.is_none()` na řádku 213 je přítomna, ale bez timeout ochrany — mrtvý proces drží kanál otevřený.

**Doporučení:**
- Přidat timeout na straně čekání na výsledek (např. 10 s)
- Při timeout zalogovat varování a resetovat `git_status_rx`

### 4. TOCTOU race při sandbox file promotion
**Soubor:** `src/app/ui/workspace/mod.rs`
**Řádky:** 254–285

```rust
if sandbox_path.exists() {
    if let Err(e) = ws.sandbox.promote_file(&rel_path) {
        // ...
    }
} else if dst.exists() {
    // Předpokládá se, že soubor byl smazán
}
```

Mezi `exists()` a `promote_file()` může být soubor externě smazán (git merge, jiný proces). `promote_file()` vrací `Err`, ale UI nezobrazuje smysluplnou chybovou zprávu v tomto případě.

**Doporučení:**
- Rozlišit v error handling, zda selhání pochází z chybějícího zdroje nebo I/O chyby
- Zobrazit uživateli konkrétní toast s příčinou

## VYSOKÉ PROBLÉMY

### 5. Zbytečné klonování v UI render loop
**Soubor:** `src/app/ui/workspace/mod.rs`
**Řádky:** 70, 246

Na každém framu se klonují struktury obsahující `String` pole:

```rust
// Řádka 70 — na každém framu
let i18n_arc = { Arc::clone(&shared.lock().unwrap().i18n) };

// Řádka 246 — clone celé Settings struktury na každém framu
let settings = shared.lock().unwrap().settings.clone();
```

Při 60 FPS jde o ~60 alokací za sekundu jen pro tyto klony.

**Doporučení:**
- Uložit `Arc<Settings>` v `AppShared` místo `Settings` (sdílení bez klonování)
- Nebo cachovat v `WorkspaceState` s dirty flagou — klonovat pouze při změně

### 6. Synchronní file I/O v UI vláknu
**Soubor:** `src/app/ui/background.rs`
**Řádky:** 88–89, 113–114

```rust
// Volá se přímo v event handler smyčce (UI thread context)
let new_content = std::fs::read_to_string(path)?;
```

Pokud je soubor velký (> 1 MB) nebo je disk pomalý (NFS, síťový svazek), UI zamrzne na dobu čtení.

**Doporučení:**
- Přesunout čtení do background vlákna přes `std::thread::spawn` nebo tokio task
- Vrátit výsledek do UI přes existující kanály

### 7. Neomezené vektory v FileWatcher
**Soubor:** `src/watcher.rs`
**Řádky:** 69–75, 149–155

```rust
pub fn try_recv(&self) -> Vec<FileEvent> {
    let mut events = Vec::new();
    while let Ok(ev) = self.receiver.try_recv() {  // Bez limitu!
        events.push(ev);
    }
    events
}
```

Pokud `notify` zdroj pošle velký burst eventů (např. git checkout s tisíci soubory), vektor nekontrolovaně roste a způsobuje memory spike + frame lag.

**Doporučení:**
- Přidat konstantu `MAX_EVENTS_PER_FRAME` (např. 500)
- Zbývající eventy ponechat v kanálu pro příští frame

### 8. Memory alokace v ProjectIndex
**Soubor:** `src/app/ui/workspace/state.rs` (voláno z fuzzy picker a search)

```rust
pub fn get_files(&self) -> Vec<PathBuf> {
    self.files.lock().unwrap().clone()  // Clone celého vektoru!
}
```

Při projektu se 100 000 soubory (každý PathBuf ~50 B) = 5 MB alokace na každé volání. Volá se z `update_filter()` při každém stisku klávesy v Ctrl+P.

**Doporučení:**
- Vrátit `Arc<Vec<PathBuf>>` — sdílení bez kopírování
- Nebo cachovat v `WorkspaceState` s invalidací při file tree změně

### 9. Absence timeout v LSP inicializaci
**Soubor:** `src/app/lsp/mod.rs`
**Řádky:** 120–186

```rust
runtime.spawn(async move {
    match client_socket_init
        .request::<Initialize>(initialize_params)
        .await   // Bez timeout!
    {
```

Pokud `rust-analyzer` nereaguje (binárka poškozená, OOM, zablokována), LSP inicializace nikdy neskončí. `initialized` flag zůstane `false`, uživatel nedostane žádnou zprávu.

**Doporučení:**
- Obalit `await` pomocí `tokio::time::timeout(Duration::from_secs(10), ...)`
- Při timeoutu nastavit error stav a zobrazit toast uživateli

## STŘEDNÍ PROBLÉMY

### 10. Off-by-one v parsování git rename výstupů
**Soubor:** `src/app/ui/background.rs`
**Funkce:** `parse_git_status()`

Pro rename/copy záznamy (formát `R old\0new`):

```rust
if matches!(x, 'R' | 'C') && i + 1 < entries.len() {
    i += 1;
    path_bytes = entries[i];
}
```

Pokud je rename záznam posledním entry a `i + 1 < entries.len()` je `false`, použije se původní `path_bytes` (zdrojová cesta místo cílové). Git status pak zobrazuje špatnou cestu.

**Doporučení:**
- Zpracovávat rename jako double-entry explicitně — přeskočit i v případě, kdy `i + 1 >= entries.len()`

### 11. Integer underflow v terminal grid indexaci
**Soubor:** `src/app/ui/terminal.rs`
**Řádky:** 245–248 (přibližně)

```rust
let line_idx = (rel_y / cell_h) as i32;
let grid_line_idx = line_idx - display_offset as i32;
// Pak se předá do get_cell() jako usize bez kontroly zápornosti
```

Pokud `display_offset` > `line_idx` (stará terminálová history), `grid_line_idx` je záporný. Konverze záporného `i32` na `usize` způsobí wrapping na velké číslo → přístup mimo paměť (UB v unsafe, panika v safe Rust).

**Doporučení:**
- Přidat: `if grid_line_idx < 0 { return None; }` před použitím jako usize

### 12. Zbytečný clone v AI diff modalu
**Soubor:** `src/app/ui/workspace/modal_dialogs/ai.rs`
**Řádky:** 43, 89–104

```rust

    // ...

```

`staged_files` je `Vec<PathBuf>` — při každém renderu se klonuje celý vektor jen proto, aby se přes něj iterovalo.

**Doporučení:**
- Nahradit za: `for rel_path in &ws.sandbox_staged_files { ... }`

### 13. Potenciální ztráta dat při shutdown race
**Soubor:** `src/ipc.rs`
**Řádky:** 71–99

Session se ukládá jako `file.tmp` → rename. Pokud aplikace crashuje přesně v okamžiku rename (extrémně raritní), session.json může chybět a .tmp zůstane na disku.

**Doporučení:**
- Při startu detekovat přítomnost `.tmp` souborů bez odpovídajícího `.json`
- Pokusit se o recovery (rename .tmp → .json)

---

### 14. Chybějící vizuální feedback stavu LSP inicializace
**Soubor:** `src/app/ui/workspace/mod.rs`
**Řádky:** 155–175

Dokud LSP neinicializuje, uživatel vidí jen chybějící autocomplete bez vysvětlení. Logika detekuje stav `lsp_client.is_none()`, ale nezobrazuje "LSP se inicializuje..." ve status baru.

**Doporučení:**
- Přidat `lsp_initializing: bool` field do `WorkspaceState`
- Zobrazit spinner nebo info text ve status baru během inicializace

## NÍZKÉ PROBLÉMY

### 15. String alokace v hot path file watchers
**Soubor:** `src/app/ui/workspace/state.rs` (a editor files handler)

```rust
let name = removed_path
    .file_name()
    .map(|n| n.to_string_lossy().into_owned())  // Heap alokace na každý event
    .unwrap_or_else(|| removed_path.to_string_lossy().into_owned());
```

Při velkém projektu s tisíci file events jde o zbytečné alokace.

**Doporučení:**
- Lazy konverze — konvertovat na String jen při zobrazení v UI

### 16. Tiché selhání při I/O chybách v IPC/session
**Soubor:** `src/ipc.rs`
**Řádky:** 75–99

```rust
let Ok(content) = std::fs::read_to_string(file_path) else {
    return vec![];  // Tichý fail, žádné logování
};
```

Pokud je `~/.config/polycredo-editor/` read-only nebo poškozený JSON, session se neobnoví bez jakéhokoli upozornění uživateli.

**Doporučení:**
- Přidat `eprintln!()` nebo propagovat toast notifikaci při parse chybě
- Rozlišit "soubor neexistuje" (normální při prvním spuštění) od "soubor nečitelný" (chyba)

### 17. Memory spike při full-text search
**Soubor:** `src/app/ui/workspace/state.rs`
**Konfigurace:** `src/config.rs` (limit 2000)

`ProjectSearch.results` drží až 2000 výsledků v paměti najednou. Každý `SearchResult` obsahuje `String` s fragmentem textu. Při hledání krátkého slova v velkém projektu = potenciálně 2000 × N bajtů.

**Doporučení:**
- Truncate výsledky na 500 v paměti, zobrazit jen aktuální stránku
- Nebo použít lazy iterátor místo `Vec`

---

### 18. Chybějící `#[must_use]` na klíčových return typech
**Soubory:** `src/app/sandbox.rs`, `src/watcher.rs`

Funkce vracející `Result<(), Error>` nemají `#[must_use]`. Volající může result ignorovat bez compiler warning.

**Doporučení:**
- Přidat `#[must_use]` atribut na funkce, kde ignorování výsledku je logická chyba
