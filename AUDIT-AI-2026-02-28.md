## Souhrn stavu

Systém je funkčně správně navržený: ReAct cyklus (REFLECTION → STATUS → PLANNING), Hierarchie pravdy, dělení scratchpad vs. facts. Instrukce jsou explicitní a zakazové podmínky u nástrojů jsou dobře napsané. Níže jsou mezery, které snižují kvalitu spolupráce.

## 1. `AiContextPayload` — chybí zásadní data

Agent dnes nezná pozici kurzoru, označený text, jméno projektu, git stav ani závislosti.

### Navrhované doplnění `types.rs`

```rust
pub struct AiContextPayload {
    // --- stávající ---
    pub open_files: Vec<AiFileContext>,
    pub build_errors: Vec<AiBuildErrorContext>,
    pub active_file: Option<AiFileContext>,
    pub memory_keys: Vec<String>,

    // --- CHYBÍ ---
    pub cursor_line: Option<usize>,
    pub cursor_col: Option<usize>,
    pub selected_text: Option<String>,
    pub project_name: String,
    pub project_root: String,
    pub git_branch: Option<String>,
    pub git_status: Vec<AiGitFileStatus>,
    pub cargo_toml_summary: Option<String>,
}

pub struct AiGitFileStatus {
    pub path: String,
    pub status: String, // "M" | "A" | "??" | "D"
}
```

### Dopad
- `cursor_line` + `selected_text` jsou kritické pro úkoly „oprav toto" nebo „refaktoruj výběr" — bez nich agent neví kde stojíš
- `git_status` zabrání editaci souborů, které uživatel právě ručně modifikoval
- `cargo_toml_summary` (jen `[package].name` + `[dependencies]`, ne celý soubor) agentovi řekne co má k dispozici bez čtení souboru

### Implementační poznámka
`generate_context()` v `mod.rs` čte `ws.editor.tabs` a `ws.build_errors`. Stačí doplnit:
- `ws.editor.cursor_pos` → `cursor_line` / `cursor_col`
- `ws.editor.selected_text()` → `selected_text`
- existující git logika ze status baru → `git_branch` + `git_status`
- lazy `std::fs::read_to_string("Cargo.toml")` + parsování → `cargo_toml_summary`

## 2. Chybí klíčové nástroje

### 2a. `ask_user` — nejdůležitější

```json
{
  "name": "ask_user",
  "description": "Ask the user a clarifying question when the task is ambiguous or requires a decision. Call this BEFORE making a guess. The user's answer will be injected as the next user message. Prevents wasted tool calls and wrong assumptions. Use 'options' for multiple-choice questions.",
  "parameters": {
    "type": "object",
    "properties": {
      "question": {
        "type": "string",
        "description": "The clarifying question to ask."
      },
      "options": {
        "type": "array",
        "items": { "type": "string" },
        "description": "Optional list of suggested answers for the user to choose from."
      }
    },
    "required": ["question"]
  }
}
```

**Proč kritické:** Bez tohoto nástroje agent při nejasném zadání buď halucinuje řešení, nebo se cyklí v nástrojích. Možnost ptát se je základní vlastnost každého kvalitního agenta.

### 2b. `announce_completion` — jak UI pozná konec?

```json
{

  "parameters": {
    "type": "object",
    "properties": {
      "summary": {
        "type": "string",
        "description": "Brief description of what was accomplished."
      },
      "files_modified": {
        "type": "array",
        "items": { "type": "string" },
        "description": "List of relative paths of files that were changed."
```

## 3. Personas nejsou Rust-aware

Dnešní persony (`Junior`, `Senior`, `Master`) jsou generické — neukotvují agenta v reáliích projektu.

### Navrhovaný společný Rust blok (přidat do `get_system_mandates` za personas)

```
RUST PROJECT STANDARDS (mandatory for all roles):
- Prefer '?' operator over '.unwrap()'. Use '.expect("reason")' only in tests or clearly unrecoverable startup paths.
- Never introduce 'unsafe' blocks without explicit user request.
- Before defining a new constant, check 'src/config.rs' — it may already exist there.
- Follow the Arc<Mutex<T>> patterns already established in the codebase. Do not introduce new synchronization primitives without need.
- After ANY code edit, validate with 'exec_in_sandbox: cargo check' before calling 'announce_completion'.
- If 'cargo check' produces errors, fix them before finishing — do not hand back broken code.
```

### Navrhované role-specifické doplnění

**Junior** — přidat:
```
RESTRICTIONS: You may NOT run 'exec_in_sandbox' with destructive commands (rm, git reset, git push).
You may NOT use 'write_file' outside the project src/ tree without explicit instruction.
When in doubt, use 'ask_user' — it is better to ask than to guess wrong.
```

**Master** — přidat:
```
For architectural changes affecting more than 3 files, first write a proposal to '.proposed_changes/PLAN.md' using 'write_file', then use 'ask_user' to get approval before implementing.
```

```markdown
## 5. Bezpečnostní mantinely (Nezlomitelná pravidla)

Tato pravidla platí bez výjimky pro VŠECHNY role a VŠECHNY úrovně reasoning:

- **NIKDY** nespouštěj `rm`, `rmdir`, `git reset --hard`, `git clean`, `git push` ani jiné destruktivní příkazy přes `exec_in_sandbox`.
```

## 5. `AiMessage` — chybí typové vyjádření tool callů

Aktuální struktura neumí zachytit strukturu tool call/result — vše je uloženo jako `String content`. Tím se ztrácí možnost auditu, replayability a debugging.

### Aktuální stav
```rust
pub struct AiMessage {
    pub role: String,        // "user" | "assistant" | "system" | "tool"
    pub content: String,     // vše jako plain text
    pub monologue: Vec<String>,
    pub timestamp: u64,
}
```

### Navrhovaná úprava
```rust
#[derive(Serialize, Deserialize, Clone, Debug)]
pub enum AiMessageContent {
    Text(String),
    ToolCall {
        name: String,
        input: serde_json::Value,
        call_id: String,
    },
    ToolResult {
        call_id: String,
        output: String,
        is_error: bool,
    },
}

pub struct AiMessage {
    pub role: String,
    pub content: AiMessageContent,
    pub monologue: Vec<String>,
    pub timestamp: u64,
}
```

**Poznámka:** Tato změna je breaking pro existující konverzace. Pokud se nechce měnit teď, stačí přidat `tool_call_name: Option<String>` a `tool_call_id: Option<String>` jako optional pole na stávající struktůře — zachová zpětnou kompatibilitu.
