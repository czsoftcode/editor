## KRITICKÁ

### K-1 · Přijímání libovolných cest ze socketu bez validace (Path Traversal)

**Soubory:** `src/ipc.rs:191-192`, `229`, `357`
**Kategorie:** Bezpečnost — path traversal / arbitrary file access

**Popis:**
Server přijímá příkazy `QUERY <cesta>`, `ADD_RECENT <cesta>` a `REGISTER <pid> <cesta>`
přes Unix socket. Cesta je okamžitě převedena na `PathBuf` a použita bez jakékoliv validace.
Útočník s lokálním přístupem k socketu (nebo jiný proces) může zadat libovolnou cestu —
například `/etc/shadow` nebo `../../../../sensitive_file` — a editor ji zpracuje jako validní
projekt.

```rust
// ipc.rs:191-192
if let Some(path_str) = line.strip_prefix("QUERY ") {
    let path = PathBuf::from(path_str);   // bez validace!
```

**Dopad:** Neautorizovaný přístup k cestám, potenciální přidání citlivých souborů do
seznamu nedávných projektů. V kombinaci s budoucími funkcemi (např. auto-otevření) by mohlo
dojít k čtení libovolných adresářů.

**Doporučení:**
1. Ověřit, že cesta je absolutní (`path.is_absolute()`).
2. Ověřit, že cesta existuje a je adresářem (`path.is_dir()`), než ji zapsat do stavu serveru.
3. Zamyslet se nad tím, zda socket musí být dostupný všem procesům uživatele, nebo jen
   procesům editoru (přes PID-based autentizaci).

## VYSOKÁ

### V-1 · `unsafe { std::env::set_var() }` v multi-threaded kontextu

**Soubor:** `src/terminal.rs:14-20`
**Kategorie:** Bezpečnost — Undefined Behavior v multi-threaded kódu

**Popis:**
Funkce `ensure_terminal_env()` volá `std::env::set_var()` z bloku `unsafe`. V Rustu ≥ 1.80
je `set_var` označeno jako `unsafe` právě proto, že modifikace environment variables v
multi-threaded programu je UB (POSIX `setenv` není thread-safe). eframe spouští GUI ve
vlastním vlákně a vytváří další vlákna pro IPC ještě před tím, než je terminál inicializován.

```rust
// terminal.rs:14-20
if !ENV_INITIALIZED.swap(true, Ordering::SeqCst) {
    unsafe {
        std::env::set_var("TERM", "xterm-256color");
        std::env::set_var("COLORTERM", "truecolor");
        std::env::set_var("PROMPT_COMMAND", "PS1='\\$ '");
    }
}
```

`AtomicBool` chrání před dvojitým nastavením, ale ne před race conditions s jinými vlákny,
která v tu chvíli čtou environment (např. standardní knihovní funkce, IPC vlákno).

**Doporučení:**
Přesunout volání `set_var` do `fn main()` ještě před vznikem jakýchkoliv vláken (před
voláním `eframe::run_native`). Tím se zcela eliminuje race condition a `unsafe` blok lze
nahradit bezpečnou variantou.

### V-2 · Command injection v průvodci novým projektem

**Soubor:** `src/app.rs:907`, `916-925`
**Kategorie:** Bezpečnost — command injection

**Popis:**
Název projektu z uživatelského vstupu (`ws.wizard_name`) je přímo předán jako argument
příkazu `cargo new <name>` nebo `composer create-project ... <name>`. Ačkoliv Rust při
volání `Command::args()` nepředá řetězec přes shell (takže klasický `; rm -rf` nefunguje),
stále platí:
- Jméno jako `../../../evil` způsobí vytvoření projektu mimo zamýšlený adresář.
- Jméno začínající pomlčkou (`-q`, `--frozen`) může být interpretováno jako příznak příkazu.

```rust
// app.rs:907, 917
let name = if project_type == ProjectType::Rust { raw_name.to_lowercase() } else { raw_name };
// ...
std::process::Command::new("cargo").args(["new", &name])
```

**Doporučení:**
Přidat validační funkci pro název projektu:
- Povolit pouze `[a-zA-Z0-9_-]` (nebo podmnožinu dle konvence cargo/composer).
- Zakázat prázdný řetězec, řetězce začínající `-`, a řetězce obsahující `/` nebo `\`.
- Zobrazit chybovou hlášku v průvodci, pokud validace selže.

### V-3 · Blokování UI vlákna při výběru souborů

**Soubor:** `src/app.rs` — použití `rfd::FileDialog`
**Kategorie:** Robustnost — blokování event loop

**Popis:**
Volání `rfd::FileDialog::new().pick_folder()` je synchronní a blokuje GUI vlákno po dobu,
dokud uživatel nevybere složku nebo nezavře dialog. Na Waylandu a některých systémech to
může blokovat eframe event loop na sekundy nebo způsobit vizuální zamrznutí.

**Doporučení:**
Použít asynchronní variantu `rfd::AsyncFileDialog` spolu s kanálem nebo `Arc<Mutex<Option<PathBuf>>>`.
Výsledek se zpracuje v dalším frame. Příklad:

```rust
let shared = Arc::clone(&self.shared);
std::thread::spawn(move || {
    let path = rfd::FileDialog::new().pick_folder();
    if let Some(p) = path {
        shared.lock().unwrap().actions.push(AppAction::OpenInNewWindow(p));
    }
});
```

## STŘEDNÍ

### S-1 · `FileWatcher::try_recv()` zahazuje všechny události kromě poslední

**Soubor:** `src/watcher.rs:57-63`
**Kategorie:** Správnost — ztráta dat

**Popis:**
Metoda drénuje celý kanál a vrací jen poslední cestu. Pokud dojde k rychlé sérii změn
(hromadná úprava více souborů), editor zaregistruje jen jednu z nich a ostatní tiše zahodí.

```rust
pub fn try_recv(&self) -> Option<PathBuf> {
    let mut last = None;
    while let Ok(path) = self.receiver.try_recv() {
        last = Some(path);   // předchozí cesty jsou ztraceny
    }
    last
}
```

**Doporučení:**
Vrátit `Vec<PathBuf>` se všemi změněnými cestami (nebo použít `HashSet` pro deduplikaci),
případně reagovat na jakoukoliv změnu invalidací celého stromu bez závislosti na konkrétní
cestě.

### S-2 · O(n·m) algoritmus vyhledávání s klonováním celého obsahu

**Soubor:** `src/editor.rs:250-271`
**Kategorie:** Výkon — škálovatelnost

**Popis:**
Při každé aktualizaci vyhledávacího dotazu se klonuje celý obsah aktivního souboru
(`t.content.clone()`) a pak se provádí lineární sken bajt po bajtu. U souboru o velikosti
1 MB a dotazu délky 5 znaků to znamená ~1 000 000 iterací na každý stisk klávesy.

```rust
let content = match self.active() {
    Some(t) => t.content.clone(),   // klonování celého souboru
    None => return,
};
// ...
while start + query_len <= content.len() {
    if content[start..start + query_len].eq_ignore_ascii_case(&self.search_query) {
        // ...
        start += 1;   // advance o 1 bajt
    }
    // ...
}
```

**Doporučení:**
1. Vyhledávat přímo přes referenci (`&t.content`), ne přes klon.
2. Zvážit inkrementální aktualizaci: pokud se dotaz prodloužil o jeden znak, filtrovat
   stávající výsledky místo opětovného skenování.
3. Pro velké soubory omezeně použít `memchr` nebo Raita/Boyer-Moore algoritmus.

### S-3 · Tiché ignorování chyb I/O operací

**Soubory:** `src/file_tree.rs:367,370`, `src/ipc.rs:50,61`, `src/watcher.rs:46,50`
**Kategorie:** Robustnost — skryté selhání

**Popis:**
Celá řada I/O operací používá vzor `let _ = std::fs::...` nebo `.ok()`, čímž tiše ignoruje
chyby. Uživatel tak nemá žádnou zpětnou vazbu o tom, proč se operace nezdařila.

```rust
// file_tree.rs:367
let _ = std::fs::create_dir(&new_path);      // selhání ignorováno

// file_tree.rs:370
let _ = std::fs::write(&new_path, "");        // selhání ignorováno

// ipc.rs:50
let _ = std::fs::create_dir_all(parent);      // selhání ignorováno
```

**Doporučení:**
Operace inicializované uživatelem (vytvoření souboru/složky, přejmenování) by měly chybu
propagovat a zobrazit uživateli v podobě toast notifikace nebo dialogu. Interní operace
(vytvoření konfiguračního adresáře) mohou selhání logovat do `eprintln!` nebo `tracing`.

### S-4 · Přijímání `EventKind::Delete` zahrnuje jen část event typů ve `FileWatcher`

**Soubor:** `src/watcher.rs:36`
**Kategorie:** Správnost — neúplné podmínky

**Popis:**
`FileWatcher` (pro sledování obsahu otevřených souborů) reaguje pouze na `is_modify()` a
`is_create()`. Smazání nebo přejmenování souboru otevřeného v editoru není zachyceno — tab
zůstane otevřený s obsahem, který na disku neexistuje, a autosave pak vytvoří soubor znovu,
což může být nežádoucí.

**Doporučení:**
Přidat obsluhu `event.kind.is_remove()`: označit tab jako "soubor smazán" a nabídnout
uživateli volbu (zavřít tab / zachovat obsah v paměti).

**Kategorie:** Udržovatelnost — složitost kódu

### N-2 · Duplikace kódu průvodce (startup wizard vs. workspace wizard)

**Soubor:** `src/app.rs`
**Kategorie:** Udržovatelnost — DRY

**Popis:**
Existují dvě místa s podobnou logikou výběru/vytváření projektu — startovací dialog
(`show_startup_dialog`) a wizard v rámci workspace (`show_workspace_wizard`). Případná změna

**Doporučení:**
Vytvořit sdílenou funkci `show_project_picker_ui(ui, shared, context: PickerContext)`
kde `PickerContext` rozlišuje kontext (startup / workspace).

### N-3 · Magická čísla rozptýlená v kódu

**Soubory:** `src/terminal.rs:11`, `src/app.rs` (různá místa)
**Kategorie:** Čitelnost — maintainability

**Popis:**
Hodnoty jako `SCROLLBAR_WIDTH: f32 = 10.0`, velikosti fontů (`14.0`, `13.0`, `15.0`),
limity (`10` nedávných projektů) jsou buď pojmenované konstanty pouze lokálně, nebo
rozptýleny jako literály přímo v kódu.

**Doporučení:**
Soustředit konfigurační konstanty do jednoho modulu `src/config.rs` nebo do struktury
`EditorConfig` s rozumnými výchozími hodnotami, aby byla příprava pro budoucí settings.toml
jednodušší.

### N-4 · Chybějící uživatelská zpětná vazba pro většinu selhání

**Soubor:** `src/app.rs`, `src/file_tree.rs`
**Kategorie:** UX — použitelnost

**Popis:**
Operace jako selhání file watcheru, nepodařené uložení session, selhání IPC socketu nebo
neexistující projekt ze session jsou buď tiše ignorovány nebo logují jen do `eprintln!`.
Uživatel nedostane žádnou informaci.

**Doporučení:**
Přidat jednoduchý toast/snackbar systém (krátkodobé notifikace v dolní části okna) pro
nekritické chyby. Kritické chyby při startu zobrazit v startovacím dialogu.

### N-5 · `session.json` a `recent.json` jsou ručně serializované

**Soubor:** `src/ipc.rs:47-76`, `394-424`
**Kategorie:** Udržovatelnost — reinventing the wheel

**Popis:**
Projekt již obsahuje závislost na `serde` (pro `eframe` persistence), ale `recent.json` a
`session.json` jsou serializovány ručním kódem s vlastním escaping mechanismem. Tento kód
je fragile (neošetřuje Unicode surrogate pairs, netestovaný edge-case: cesty obsahující `]`).

**Doporučení:**
Použít `serde_json::to_string` / `serde_json::from_str` pro oba soubory. Závislost
`serde_json` je již tranzitivně přítomna přes `eframe`.

## Pozitivní hodnocení

Následující aspekty kódu jsou na dobré úrovni a zaslouží zmínku:

- **Správné použití RAII:** `IpcServer`, `FileWatcher` a `_watcher` field správně uklízí
  zdroje přes `Drop`.
- **Bezpečné sdílení stavu:** `Arc<Mutex<WorkspaceState>>` a `Arc<Mutex<AppShared>>` jsou
  správně použity pro komunikaci mezi viewporty bez race conditions.
- **Build pipeline mimo UI vlákno:** `cargo build` a podobné příkazy jsou spouštěny ve
  vlastních vláknech (`std::thread::spawn`), výsledky předávány kanálem — UI nikdy neblokuje.
- **Modulární struktura:** Oddělení `editor.rs`, `file_tree.rs`, `terminal.rs`,
  `watcher.rs`, `ipc.rs` je logické a přehledné.
- **Atomická zápis souborů:** `save_recent` a `save_session` používají vzor `write tmp →
  rename` pro atomický zápis bez rizika poškozené partial write.
- **Správné zrušení close eventu:** Sekundární viewporty zachycují `close_requested` a
  odesílají `CancelClose` + akci do fronty místo přímého zavření — správná integrace s
  eframe viewport API.
