## Nálezy (seřazeno podle závažnosti)

### K-1 · Riziko přepsání binárního/nečitelného souboru chybovým textem
**Soubor:** `src/app/ui/editor/tabs.rs:80`, `src/app/ui/editor/tabs.rs:105`  
**Typ:** Logika / integrita dat

Při selhání `read_to_string()` se otevře běžný editovatelný tab s obsahem `Error reading file: ...`, ale se stejnou `path` jako původní soubor. Pokud uživatel následně uloží, aplikace zapisuje obsah tabu zpět do původní cesty.

**Dopad:** možná korupce původního souboru (zejména ne-UTF8/binary soubory mimo whitelist obrázků).  
**Doporučení:** při selhání čtení otevřít read-only error view bez navázání na původní cestu, nebo fallbacknout na binární režim.

### V-1 · Watcher filtr vylučuje sandbox, ale zbytek kódu na sandbox eventech závisí
**Soubory:** `src/watcher.rs:113`, `src/app/ui/workspace/state.rs:265`, `src/app/ui/background.rs:77`, `src/app/ui/background.rs:110`  
**Typ:** Logika / výkon

`ProjectWatcher` explicitně ignoruje komponenty `.polycredo`, `history`, `sandbox`, ale workspace zároveň přidává watch na `sandbox.root` a event handler čeká sandbox eventy (dirty flag + auto diff).

**Dopad:** sandbox změny se nepropíšou okamžitě, části logiky (`pending_ai_diff`) se nemusí spouštět, systém padá na periodické full scanování.  
**Doporučení:** sjednotit pravidla: buď sandbox nesledovat a odstranit závislou logiku, nebo sandbox nevynechávat ve filtru.

### V-2 · Trvalý full scan sandboxu každé 3 sekundy + O(n²) deduplikace
**Soubory:** `src/app/ui/workspace/mod.rs:53`, `src/app/ui/workspace/mod.rs:54`, `src/app/sandbox.rs:109`, `src/app/sandbox.rs:171`  
**Typ:** CPU / I/O / paměť

Obnova staged cache běží periodicky i bez změn (`elapsed >= SANDBOX_STAGED_REFRESH_MS`). `get_staged_files()` prochází celý sandbox i projekt, navíc v části detekce delete dělá deduplikaci přes `Vec::contains` (kvadratická složitost).

**Dopad:** konstantní disková zátěž a CPU overhead u větších projektů, i v nečinnosti editoru.  
**Doporučení:** refresh jen při eventech/dirty epoch, deduplikace přes `HashSet<PathBuf>`.

### V-3 · Synchronní čtení souborů v UI vlákně při FS eventech
**Soubory:** `src/app/ui/background.rs:89`, `src/app/ui/background.rs:113`, `src/app/ui/background.rs:122`, `src/app/ui/background.rs:144`  
**Typ:** Latence UI / CPU / I/O

V handleru změn (běží z `render_workspace` každým framem) se při událostech provádí `std::fs::read_to_string` přímo v UI vlákně.

**Dopad:** při větších souborech nebo burstu eventů dochází k viditelnému lagování/zamrzání UI.  
**Doporučení:** přesunout čtení + diff vstupy do background workeru a do UI vracet jen výsledek.

---

### V-4 · Full-text syntax highlighting se přepočítává při každém renderu editoru
**Soubory:** `src/app/ui/editor/render_normal.rs:88`, `src/highlighter.rs:58`, `src/app/ui/editor/search.rs:222`  
**Typ:** CPU / paměť

Layouter volá syntax highlighting nad celým textem a následně ještě lineární aplikaci search zvýraznění přes sekce. U dlouhých souborů je to náročné na CPU i alokace.

**Dopad:** výrazné zpomalení psaní/scrollu ve velkých souborech.  
**Doporučení:** line-based cache/incrementální highlighting; search highlight řešit bez průchodu všech sekcí každé renderování.

### V-5 · LSP `didChange` posílá celý dokument při každé změně
**Soubory:** `src/app/ui/editor/render_normal.rs:167`, `src/app/ui/editor/render_markdown.rs:254`, `src/app/lsp/mod.rs:245`  
**Typ:** CPU / paměť / IPC

Při každé editaci se klonuje celý `tab.content` a posílá jako full-sync změna.

**Dopad:** vysoké nároky na paměť a komunikaci s LSP serverem u velkých souborů, možné špičky latence při psaní.  
**Doporučení:** throttle/debounce notifikací, případně incremental range changes.

### S-1 · Markdown split view klonuje celý obsah a reparsuje preview při každém frame
**Soubory:** `src/app/ui/editor/render_markdown.rs:211`, `src/app/ui/editor/markdown.rs:10`  
**Typ:** CPU / paměť

V markdown režimu se každý frame klonuje celý text a preview se znovu renderuje přes `CommonMarkViewer`.

**Dopad:** rychlý růst CPU spotřeby a alokací při větších markdown souborech.  
**Doporučení:** invalidovat preview jen při změně obsahu (dirty flag), držet cache per tab.

---

### S-2 · Project search dělá vysoké množství alokací na každém řádku
**Soubor:** `src/app/ui/search_picker.rs:213`  
**Typ:** CPU / paměť

Vyhledávání vytváří `line.to_lowercase()` pro každý řádek každého souboru. Současně se soubory načítají celé do paměti (`read_to_string`).

**Dopad:** vysoké GC/alokační zatížení a pomalé hledání na velkých repozitářích.  
**Doporučení:** používat case-insensitive matching bez alokace na každém řádku, zavést velikostní limit souborů a skip binárních dat.

---

### S-3 · Fronta watcher eventů je neomezená a jedno zpracování může vyčerpat frame budget
**Soubory:** `src/watcher.rs:69`, `src/watcher.rs:149`, `src/watcher.rs:50`, `src/watcher.rs:129`  
**Typ:** CPU / paměť / odezva UI

Použité `mpsc::channel` je neomezené, callback do něj bez omezení posílá eventy a `try_recv`/`poll` je v jednom průchodu vybere všechny.

**Dopad:** při burstu (např. masivní git operace) mohou vznikat velké špičky paměti i dlouhé bloky v UI ticku.  
**Doporučení:** limit eventů na frame + deduplikace (`HashSet<PathBuf>`) + backpressure.

---

### S-4 · Nastavení tématu/fontu se aplikuje každý frame
**Soubory:** `src/app/mod.rs:425`, `src/settings.rs:137`  
**Typ:** CPU

`settings.apply(ctx)` běží v každém `update()`, i když se hodnoty nezměnily.

**Dopad:** zbytečné přepisování visuals/style a práce navíc v hot path UI loopu.  
**Doporučení:** aplikovat jen při změně settings (dirty flag / hash poslední aplikované konfigurace).

---

### S-5 · Detekce změněného tabu canonicalizuje cestu všech tabů
**Soubory:** `src/app/ui/background.rs:39`, `src/app/ui/editor/files.rs:14`, `src/app/ui/editor/files.rs:17`  
**Typ:** CPU / I/O

Při každém file eventu se hledá odpovídající tab přes `canonicalize()` každé tab cesty.

**Dopad:** zbytečné filesystem volání a latence při větším počtu otevřených tabů.  
**Doporučení:** držet mapu canonical path -> tab index (aktualizovat při open/close/rename).
